# Архитектура системы

## 1. Общая информация

Проект: labyrinth-support-api  
Жанр: 2D Лабиринт / Roguelite / Extraction  
Клиент: Unity 2D Demo / test_client.html  
Backend: Flask + MySQL 8.0  

## 2. Назначение системы

Данная система представляет собой backend-сервер, предназначенный для поддержки игрового процесса.

Основные задачи сервера:

- регистрация и авторизация пользователей;
- хранение данных игроков;
- управление игровыми сессиями (вылазками);
- обработка результатов игры;
- логирование игровых событий.

Сервер выступает как центральный компонент, отвечающий за хранение и обработку состояния игры.

---

## 3. Общая архитектура

Система построена по клиент-серверной архитектуре.

### Схема взаимодействия:

Client → API → Business Logic → Database

---

## 4. Слои системы

### 4.1 Client

Клиентом является:
- тестовый HTML-интерфейс (`test_client.html`);
- потенциально Unity-приложение.

Клиент:
- отправляет HTTP-запросы;
- передаёт данные игрового процесса;
- получает ответы от сервера.

---

### 4.2 API (Flask)

API реализовано на Flask и включает:

- маршруты (routes);
- контроллеры (controllers).

Функции API:

- приём HTTP-запросов;
- проверка входных данных;
- проверка JWT-токена;
- передача данных в бизнес-логику;
- возврат ответа клиенту.

---

### 4.3 Business Logic (Services)

Бизнес-логика реализована в слое `services`.

Основные модули:

- auth_service — регистрация и авторизация;
- profile_service — работа с профилем игрока;
- run_service — управление вылазками;
- events_service — обработка игровых событий.

Этот слой отвечает за:

- правила игры;
- расчёт наград;
- изменение состояния игрока.

---

### 4.4 Data Layer (MySQL)

База данных используется для хранения:

- пользователей;
- ролей;
- прогресса игрока;
- вылазок;
- результатов;
- событий.

Доступ к данным осуществляется через модели и репозитории.

---

## 5. Аутентификация

В системе используется JWT-аутентификация.

Процесс:

1. Пользователь выполняет login
2. Сервер выдаёт токен
3. Клиент отправляет токен в каждом запросе
4. Сервер проверяет токен

Формат заголовка:

Authorization: Bearer <token>

---

## 6. Основной игровой сценарий

1. Пользователь регистрируется или логинится  
2. Получает JWT-токен  
3. Начинает вылазку (`run/start`)  
4. Отправляет игровые события (`events`)  
5. Завершает вылазку (`run/finish`)  
6. Получает награды  
7. Проверяет профиль (`profile`)  

---

## 7. Обработка вылазки

При завершении вылазки сервер:

- проверяет корректность данных;
- определяет результат (успех или смерть);
- рассчитывает награды;
- обновляет прогресс игрока;
- сохраняет результат в базе.

---

## 8. Преимущества архитектуры

- чёткое разделение логики;
- масштабируемость;
- независимость клиента и сервера;
- возможность подключения разных клиентов.

---

## 9. Вывод

Реализованная архитектура соответствует требованиям клиент-серверных систем и позволяет поддерживать игровой процесс, хранить состояние игроков и обрабатывать игровые события.
